# ADR-0001. 모듈러 모놀리스 + DDD-lite 컨텍스트 패키지 구조

- 상태: 수용(Accepted)
- 일자: 2026-07-19

## 맥락

통합인증 데모는 배포 단위가 하나인 단일 백엔드(Spring Boot)다. 규모는 작지만,
인증이라는 도메인은 시간이 지나면서 정책이 늘어나기 쉽다(가입 정책, 소셜 연동 확장,
본인인증 방식 교체 등). 코드가 커질 때 "어디에 무엇을 두는가"가 흔들리지 않도록
초기부터 패키지 경계를 정해 둔다.

동시에 이 프로젝트는 **데모**다. 완전한 DDD 전술 설계(애그리거트마다 저장소 구현체,
ORM 엔티티와 도메인 엔티티 분리 등)를 모두 적용하면 배보다 배꼽이 커진다. 검증된
구조를 가져오되 **데모에 맞게 축소**한다.

## 결정

**모듈러 모놀리스 안에서 DDD를 일부 차용한 컨텍스트 패키지 구조**를 쓴다.

### 패키지 최상위 구분

```
com.example.demo
├── module/           # 도메인 요구사항을 담는 컨텍스트 그룹
│   └── <context>/
└── common · config   # 도메인과 무관한 공용·설정 (module 을 참조하지 않음)
```

- `module`은 `common`·`config`를 참조할 수 있으나, 그 반대는 금지한다.
  의존은 항상 안정적인 하부(공용·인프라)를 향한다.

### 컨텍스트 내부 구조

컨텍스트 패키지 내부는 두 축으로 나눈다.

- `application`: **유스케이스(요구사항)** 단위 — `Controller`, `Service`, `QueryService`
- `domain`: **애그리거트** 단위 — 엔티티(애그리거트 루트), 저장소(Repository)
- `infrastructure`(필요할 때만): 외부 연동 어댑터 등

컨트롤러는 개념상 프레젠테이션이지만 별도 `presentation` 패키지를 두지 않고
해당 기능의 `application`에 함께 둔다. 레이어 구분은 폴더가 아니라 클래스 역할·
애너테이션(`@RestController` 등)으로 드러낸다.

### 데모에서의 컨텍스트 범위

데모의 도메인이 "통합 인증" 하나에 가깝기 때문에, 컨텍스트를 잘게 쪼개지 않고
**`identity`(사용자·계정 통합) 컨텍스트 하나로 시작**하기로 확정했다. 본인인증/OAuth
같은 인증 수단 처리도 당분간 `identity` 안에 둔다 — 별도 `verification` 컨텍스트로
분리하는 것은 필요가 분명해질 때 다시 판단한다.

컨텍스트가 여럿이 되어 서로 협력할 때만 상류-하류(OHS/ACL) 규칙을 도입한다.
컨텍스트가 하나뿐인 동안에는 ACL/OHS를 **미리 만들지 않는다** — 필요해질 때 도입한다.

## 결과

- **장점**: 도메인 지식이 `module` 안에 모여 공용 코드로 새지 않는다. 인증 정책이
  늘어도 컨텍스트 경계 안에서 확장된다.
- **비용**: 컨텍스트가 하나뿐인 초기에는 다소 과한 구조로 보일 수 있다. 이는 의도적인
  선행 투자이며, 규칙을 미리 정해 두는 것이 나중에 정리하는 것보다 싸다.
- **확정**: 데모 컨텍스트는 **`identity` 하나**로 시작한다(2026-07-19). 인증 임시데이터·
  RFT 저장도 이 컨텍스트 안에 둔다([ADR-0005](0005-verification-temporary-data-storage.md)).
