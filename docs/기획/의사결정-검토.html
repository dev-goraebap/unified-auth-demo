확정 이미 결정됨 (ADR 반영) 방금 확정 이번에 정함 검토 필요 선택 대기

✅ 확정된 결정

세션 전략 — JWT Access + Refresh Token 방금 확정

ADR-0006

짧은 수명의 Access Token(무상태 검증) + 서버 저장 Refresh Token(회전·폐기 지원). 프론트/백 분리 구조에 맞춤. → RFT 저장소가 필요해짐(아래 4번과 연결).

백엔드 구조 — 모듈러 모놀리스 + DDD-lite

ADR-0001

module/<context> 아래 application(유스케이스)/domain(애그리거트)/infrastructure. 데모라 컨텍스트는 최소화.

데이터 접근 — 명령 JPA / 조회 jOOQ(동적)

ADR-0002

명령은 JPA로 도메인 불변식 보호, 조회는 jOOQ 동적 타입(코드젠 없음). MyBatis 미채택.

DB — PostgreSQL 로컬 도커

ADR-0003

도커 postgres:16에 데모 전용 DB unified_auth 생성. 접속정보는 커밋 제외(application-local.properties).

본인인증 — Provider 추상화 (Mock ↔ 실제)

ADR-0004

인터페이스 하나로 감싸 auth.pass.provider=mock|real 설정으로 교체. Mock은 이름+생년월일+성별 해시로 DI를 결정적으로 생성.

🟡 정해야 할 것 — 골라주시면 반영합니다

1

가입 정책

신원(DI)을 언제 확보하느냐. 흐름의 뼈대가 됩니다.

A본인인증 먼저 → DI 확보 후 로그인 수단 등록권장
B소셜로 먼저 가입하고 나중에 PASS로 실명연결
2

기존 계정에 새 소셜 연결 시

로컬 회원이 소셜로 처음 들어와 PASS 통과 후 DI가 일치할 때. → 디자인할 화면이 달라집니다.

A확인 화면 표시 — "○○○님 계정에 카카오를 연결할까요?"권장
B말없이 자동 연결 (본인인증으로 이미 안전하지만 사용자가 놀랄 수 있음)
3

Mock 본인인증 화면 방식

테스트용 본인인증을 어떻게 띄울지.

A백엔드 리다이렉트 방식 — 실제 PASS처럼 별도 인증창 이동→콜백. 실전 코드 경로를 최대한 살림권장
BAngular 안의 간단한 입력 모달 (가장 빠르지만 리다이렉트 흐름 생략)
4

임시데이터 · Refresh Token 저장

OAuth state, PASS 결과 임시보관, 그리고 확정된 RFT(ADR-0006)를 어디에 둘지.

A임시데이터 = verification 테이블 + TTL, RFT = 별도 refresh_token 테이블권장
B임시데이터는 인메모리(TTL 캐시), RFT만 테이블 (데모 한정 단순화)
5

스키마 관리 도구

jOOQ 동적 타입은 코드젠이 없어 스키마 소스 관리가 중요.

A초기엔 JPA ddl-auto로 빠르게 → 스키마 안정 후 Flyway 도입권장
B처음부터 Flyway로 스키마 버전 관리
6

바운디드 컨텍스트 개수

인증 도메인을 몇 덩어리로 나눌지.

Aidentity 하나로 시작 (데모 규모에 충분, 필요 시 분리)권장
Bidentity + verification 둘로 분리
참고 — 위 6개는 모두 권장안(A)이 있습니다. "전부 권장안대로" 하셔도 되고, 바꾸고 싶은 항목만 알려주셔도 됩니다. 정해지면 ADR-0005를 수용으로 확정하고 스키마·API 설계로 넘어갑니다.
