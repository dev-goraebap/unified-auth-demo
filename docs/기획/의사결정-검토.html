아키텍처 결정

백엔드 구조 — 모듈러 모놀리스 + DDD-lite

ADR-0001

module/identity 하나로 시작. application(유스케이스)/domain(애그리거트)/infrastructure.

데이터 접근 — 명령 JPA / 조회 jOOQ(동적)

ADR-0002

명령은 JPA로 불변식 보호, 조회는 jOOQ 동적 타입(코드젠 없음). MyBatis 미채택.

DB — PostgreSQL 로컬 도커

ADR-0003

데모 전용 DB unified_auth. 접속정보는 application-local.properties(gitignore).

스키마 관리 — 초기 ddl-auto → Flyway

ADR-0003

데모 초기엔 JPA ddl-auto=update로 빠르게, 스키마 안정 후 Flyway 전환.

세션 전략 — JWT Access + Refresh Token

ADR-0006

짧은 Access Token(무상태) + 서버 저장 RFT(회전·폐기). 프론트/백 분리에 맞춤.

임시데이터 · RFT 저장 — 테이블 2종

ADR-0005

verification(OAuth state·본인인증 결과 임시보관 + TTL) + 별도 refresh_token. 둘 다 identity 소속.

본인인증 결정

Provider 추상화 (Mock ↔ 실제)

ADR-0004

auth.pass.provider=mock|real 설정으로 교체. Mock은 이름+생년월일+성별 해시로 DI를 결정적 생성.

연동 방식 — 애그리게이터 패턴 채택

ADR-0004

Bootpay·PortOne式: 프론트 SDK가 인증창 오픈 → 식별자 반환 → 백엔드 verify(reference) REST 조회. KCP식 백엔드 콜백·복호화는 미채택. Mock도 이 모양으로 구현.

정책 결정

가입 정책 — 본인인증 우선

PASS 인증 먼저 → DI 확보 → 로그인 수단(로컬/소셜) 등록.

기존 계정에 소셜 연결 — 확인 화면 표시

DI 일치 시 자동 연결하지 않고 "○○○님 계정에 연결할까요?" 확인 화면을 한 번 노출.

본인인증(애그리게이터 패턴) Mock 흐름
  1. 프론트: "가짜 인증창"에서 이름·생년월일·성별 입력 → 백엔드에 임시저장하고 가짜 식별자(reference) 수령
  2. 프론트 → 백엔드: 그 식별자를 서버로 전달
  3. 백엔드: MockVerificationProvider.verify(reference) → 저장된 입력값으로 DI/CI 합성 → 가입·연결·로그인 진행
실제 전환 시 프론트는 SDK 호출부만, 백엔드는 verify() 구현만 교체(Bootpay/PortOne). 상세: docs/기획/본인인증-방식비교.html
