통합인증 데모 — 설계 정리

PASS 본인인증 · 소셜 로그인 · 로컬(ID/PW) 로그인을 하나로 묶는 인증 구조

Angular + Spring Boot DI 앵커 모델 Mock PASS 인증

◎이 문서는

실제 PASS 본인인증은 대행사와 계약해야 테스트가 가능하기 때문에, PASS 부분을 Mock(가짜 인증)으로 대체하면서도 실제 연동 시 코드 변경이 최소화되도록 설계한 내용을 정리한 문서입니다.

CI vs DI 도메인 모델 핵심 플로우 소셜 미가입자 진입 Mock PASS & DI 생성 교체 지점 남은 결정사항

1개념 정리 — CI vs DI

구분범위용도
CI Connecting Information 개인당 1개, 모든 서비스 공통 서비스 간 동일인 식별
DI Duplication Information 개인당 1개지만 서비스마다 다름 한 서비스 내 중복가입 방지
한 서비스 안에서 "이 계정들은 동일인"임을 판별하는 앵커(anchor)로는 DI가 정답입니다. 나중에 다른 서비스와 연동할 가능성을 대비해 CI도 함께 저장해 둡니다.

2도메인 모델 (DI 중심)

한 사람 = 하나의 DI. 그 DI에 로컬 계정 최대 1개 + 소셜 계정 여러 개가 붙습니다.

User (사람 = DI 하나)
 ├─ di          (unique, PASS에서 획득)   ← 앵커
 ├─ ci          (선택, 크로스서비스 대비)
 ├─ name, birth, gender, phone           (PASS 인증 결과)
 │
 ├─ LocalCredential   (0..1)   ← 로컬 계정 최대 1개
 │    ├─ login_id (unique)
 │    └─ password_hash
 │
 └─ SocialAccount     (0..N)   ← 소셜 여러 개
      ├─ provider          (kakao | naver | google)
      ├─ provider_user_id
      └─ unique(provider, provider_user_id)  ← 한 소셜계정은 한 User에만

테이블로는 users, local_credentials, social_accounts 3개가 핵심입니다.

33가지 핵심 플로우

① 신규 가입 (PASS 우선)

PASS 본인인증 → DI 획득
↓ DI로 users 조회
없음 → User 신규 생성 → 로그인 수단(로컬 or 소셜) 등록
있음 → "이미 가입된 회원. 로그인 수단을 추가하시겠어요?"

② 로그인

로컬: id / pw → local_credentials → User
소셜: OAuth 콜백 → (provider, provider_user_id) → social_accounts → User
↓ 매칭되는 소셜계정이 없으면
"본인인증 후 계정 연결 필요" → PASS 플로우로 유도

③ 계정 연결 (로그인된 상태에서)

  • 소셜 추가 연결 / 로컬 자격증명 설정
  • 붙이려는 소셜계정이 다른 DI에 이미 물려 있으면 거부 (중복 방지)

4소셜 로그인 미가입자가 들어오면?

"회원가입 안 하고 소셜 로그인"이라는 상태 자체가 없습니다. DI 앵커 모델에서는 DI 없는 계정이 존재할 수 없어서, 소셜로 처음 온 사람도 PASS를 한 번은 거칩니다.

"카카오로 로그인" 클릭 → OAuth 성공 → (kakao, 123) 획득
↓ social_accounts 조회
매칭 있음 → 바로 로그인 (그 User의 DI로 세션 발급) ★ 2번째부터 이 경로
매칭 없음 → PASS 인증 유도 → DI 획득 → users를 DI로 조회
DI 있음 → 기존 회원! 이 카카오계정을 기존 User에 자동 연결 → 로그인
DI 없음 → 신규 User 생성 + 카카오계정 연결 → 로그인 (= 소셜 가입)
① PASS는 처음 한 번만. 소셜계정이 아직 어떤 DI에도 안 붙어 있을 때만 본인인증을 요구합니다. 한 번 연결되면 다음부터는 소셜 매칭으로 바로 로그인 → 매번 인증하지 않습니다.
② 연결은 반드시 DI로만 — 이메일 매칭 금지. "카카오 이메일이 기존 계정과 같으니 자동 연결"은 계정 탈취 통로가 됩니다. 반면 DI 기반 자동 연결은 안전합니다. PASS가 이미 "실명상 동일인"을 증명했기 때문입니다.

사용자 입장 시나리오

상황경험
완전 신규가 카카오 클릭카카오 인증 → PASS 1회 → 가입 완료 & 로그인
로컬로 가입했던 사람이 카카오 클릭카카오 인증 → PASS 1회 → "기존 계정에 연결됨" → 로그인
카카오 연결 끝난 사람이 카카오 클릭카카오 인증 → 바로 로그인 (PASS 없음)

5Mock PASS & DI 생성

실제 PASS가 없으니 Mock이 DI를 "결정적으로(deterministic)" 생성합니다. 테스터가 Mock 인증창에 이름 · 생년월일 · 성별을 입력하면 그 값에서 DI를 계산합니다.

identityCore = 이름 + 생년월일 + 성별            // 이 사람이 누구인가
DI = Base64( SHA-256( SITE_SALT   + identityCore ) )   // 88자 문자열
CI = Base64( SHA-256( COMMON_SALT + identityCore ) )
  • 결정적 → 같은 사람 정보면 항상 같은 DI → "이미 가입 / 기존계정 연결" 분기 재현 가능
  • 이름만 바꾸면 다른 DI → 신규가입 테스트
  • SITE_SALT는 서비스 고정 상수(사이트별로 DI가 갈리는 성질을 흉내)
휴대폰번호는 DI 재료에서 제외. 실제 PASS에서 DI는 주민번호 기반이라 번호가 바뀌어도 변하지 않습니다. 그래서 Mock도 DI는 이름/생년월일/성별로만 만들고, 휴대폰번호는 별도 "연락처 정보"로 저장합니다.
데모 한계: 이름·생년월일·성별이 같으면 동일 DI로 취급됩니다(실제로는 주민번호로 구분). 신경 쓰이면 Mock 화면에 "주민번호 뒷자리" 같은 식별필드를 하나 더 받아 identityCore에 넣으면 충돌이 거의 사라집니다.

실제 DI 모양 흉내내기

실제 DI는 64바이트 → Base64로 약 88자입니다. Mock도 같은 길이/포맷으로 만들면 users.di VARCHAR(88) 스키마와 조회 로직을 진짜 PASS와 동일하게 유지할 수 있습니다.

Mock 인증창 위치

방식설명
(A) 백엔드 HTML 폼 페이지 ← 추천 실제 PASS의 "외부 인증창 리다이렉트 → 콜백" 흐름을 그대로 흉내. 실전 연동 코드 경로를 최대한 살림.
(B) Angular 테스트 입력 모달 가장 빠르지만 리다이렉트/콜백 흐름은 건너뜀.

6교체 지점은 딱 한 곳

DI를 소비하는 쪽(가입/로그인/연결 로직)은 Mock인지 진짜인지 전혀 몰라야 합니다. 인터페이스 하나로 감싸고 설정 한 줄로 스위칭합니다.

interface IdentityVerificationProvider {
    VerificationResult verify(...);   // { di, ci, name, birth, gender, phone }
}

@ConditionalOnProperty("auth.pass.provider" = "mock")
class MockVerificationProvider   → 입력값 해시로 DI 생성

@ConditionalOnProperty("auth.pass.provider" = "real")
class PassVerificationProvider   → 나중에 실제 연동
설정 한 줄(auth.pass.provider = mock | real)로 교체. 실제 PASS 계약 후에는 provider 구현체만 갈아끼우면 끝입니다.

7아직 정하지 않은 것

코드 뼈대(API 목록/인터페이스)를 확정하기 전에 결정이 필요한 항목입니다.

Q1. 가입 정책

"PASS 인증 필수 → DI 확보 후 로그인 수단 등록" 방식으로 확정할지. (이 방식이 DI 앵커 모델과 가장 깔끔함)

Q2. 자동 연결 vs 확인 화면

로컬 회원이 소셜로 처음 들어와 PASS 통과 후 DI가 일치할 때, 말없이 자동 연결할지 / "○○○님 계정에 카카오를 연결할까요?" 확인 화면을 한 번 보여줄지.

Q3. 세션 전략

프론트/백 분리 구조 → JWT(access/refresh) vs 세션 쿠키 중 선택.
