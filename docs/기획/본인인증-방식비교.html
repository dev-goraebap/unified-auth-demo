KCP 직접 · Bootpay · PortOne 비교

본인인증(휴대폰/PASS) 연동은 크게 두 가지 패턴으로 갈립니다. 어느 쪽으로 갈지에 따라 데모의 Mock 설계가 달라져, 세 서비스를 정리했습니다.

큰 그림 — 두 가지 패턴

애그리게이터 패턴

Bootpay · PortOne

프론트 SDK가 인증창을 열고, 백엔드는 조회 한 번

1프론트
SDK가 인증창(팝업) 오픈 → 사용자 인증
2프론트
성공 시 식별자만 반환(receipt_id/identityVerificationId) → 서버로 전달
3백엔드
식별자로 REST 단건조회 → CI/DI·이름 등 획득

✅ 콜백/HTML 서빙 없음 · 복호화 없음 · 백엔드 가벼움

직접 연동 패턴

KCP

백엔드가 파라미터·콜백·복호화를 모두 처리

1백엔드
거래등록 API → call_url·키 획득
2프론트
그 URL로 form submit → 인증창 오픈 → 사용자 인증
3백엔드
Ret_URL 콜백 수신(직접 서빙) → 결과조회 API
4백엔드
암호화 데이터 복호화(ENC_KEY) → CI/DI 추출

⚠️ 콜백 엔드포인트 + 복호화 배선 필요 · 백엔드 무거움

결론: Bootpay와 PortOne은 “동일 패턴”입니다 둘 다 SDK가 인증창 오픈 → 식별자 반환 → 백엔드 REST 조회로 CI/DI 획득 3단계가 똑같습니다. 이름만 다를 뿐(receipt_id ↔ identityVerificationId), 구조는 갈아끼우기 쉬운 수준으로 유사합니다.

데모 관점에서의 시사점

Mock은 “애그리게이터 패턴” 모양으로 만들면 Bootpay든 PortOne이든 전환 가능 우리 백엔드 추상화 IdentityVerificationProvider.verify(reference)에서 reference = 식별자(receipt_id / identityVerificationId) 로 두면, Mock은 “가짜 식별자 → 백엔드가 저장해둔 입력값으로 DI 합성” 한 줄이면 됩니다. 실제 전환 시 프론트는 SDK 호출만 바꾸고, 백엔드는 provider 구현만 교체합니다.
유의사항
  • DI 제공 여부는 대행사별로 다름 — 일부 채널(예: KG이니시스 통합인증)은 DI를 안 줄 수 있음. 실제 연동 시 채널 확인 필요.
  • Bootpay certificate 응답의 정확한 필드 스키마, PortOne V2 phoneNumber/operator 필드는 문서 로그인 게이트로 일부 확인 불가 — 계정으로 문서 확인 필요.
  • 어느 경우든 실제 CI/DI를 받으려면 대행사 계약/이용료가 발생 (데모는 Mock으로 대체).

정리

  • Bootpay ≈ PortOne — 같은 애그리게이터 패턴. 둘 중 무엇을 골라도 데모 구조·Mock은 동일.
  • KCP — 직접 연동. 백엔드에 콜백·복호화 배선이 추가로 필요(무거움).
  • 데모 권장: 애그리게이터 패턴으로 Mock 설계 → 나중에 Bootpay/PortOne 중 실제 계약한 쪽으로 provider만 교체.
